## Goal

Replace the `cft` CLI's placeholder `LiveLayer` (currently all `notImplemented` stubs) with real implementations backed by the `tunnel-sdk/effect` services. This connects the CLI to the real Cloudflare API.

## Context

The `cft` CLI package (`packages/cft/`) already uses Effect. It defines its own service interfaces in `src/services.ts` (`QuickTunnelService`, `TunnelApiService`, `IngressService`, `RouteService`, `DnsService`, `VNetService`, `ConfigService`, `AuthService`) and has a `live-layer.ts` that provides stubs that all return `notImplemented`.

The goal is to create **adapter layers** that bridge the cft service interfaces to the `tunnel-sdk/effect` services. Some cft services map 1:1 to SDK services; others (like `ConfigService`, `AuthService`) are cft-specific and stay as stubs or get their own implementation.

## What to do

### 1. Read and understand the cft service interfaces

Read `packages/cft/src/services.ts` carefully. Map each cft service method to the corresponding SDK service method:

| `TunnelApiService.create(name, opts?)` | `TunnelOperations.create(name, opts?)` | Map `TunnelInfo` â†’ `TunnelInfo` (cft type) |

### 2. Update `packages/cft/src/live-layer.ts`

Replace the stubs with real adapter implementations. Import from `tunnel-sdk/effect`:

```ts
import { Effect, Layer } from "effect"
import {
  TunnelOperations, IngressManager, DnsManager, RouteManager, VNetManager,
  TunnelProcessService, CloudflaredBinary, LiveLayerFromEnv, expose,
} from "tunnel-sdk/effect"
```

For services that map cleanly (VNetService, etc.), create adapter layers:
```ts
const VNetServiceLive = Layer.effect(
  VNetService,
  Effect.gen(function*() {
    const vnets = yield* VNetManager
    return {
      create: Effect.fn(function*(name, opts?) {
        yield* vnets.create(name, { default: opts?.isDefault, comment: undefined })
      }),
      list: Effect.fn(function*() {
        const results = yield* vnets.list()
        return results.map(v => ({ name: v.name, isDefault: v.isDefault }))
      }),
      delete: Effect.fn(function*(name) {
        yield* vnets.delete(name)
      }),
    }
  })
)
```

For `TunnelApiService.run` and `TunnelApiService.stop`, you'll need to manage a running tunnel reference (possibly via `Ref` or `FiberHandle`).

### 3. Update `packages/cft/package.json`

Add `tunnel-sdk` as a dependency (workspace link):
```json
{
  "dependencies": {
    "tunnel-sdk": "workspace:*"
  }
}
```

### 4. Map errors

The cft uses `CftError` (from `packages/cft/src/errors.ts`). SDK services throw `TunnelApiError`, `TunnelAuthError`, etc. The adapter layers need to catch SDK errors and wrap them in `CftError`:

```ts
Effect.catchTags({
  TunnelApiError: (e) => CftError.ApiError({ message: `API error ${e.status}` }),
  TunnelAuthError: (e) => CftError.AuthError({ message: e.message }),
  TunnelNotFoundError: (e) => CftError.NotFoundError({ message: `Tunnel not found: ${e.tunnelRef}` }),
})
```

### 5. Keep ConfigService and AuthService as stubs (or implement separately)

These are CLI-specific services that don't have SDK counterparts. They can stay as `notImplemented` stubs for now, or be implemented as a follow-up.
