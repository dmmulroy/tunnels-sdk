## What to implement
Track running tunnel processes so `stop` can kill them and `getLogs` can read their log streams.

### State management approach
Use `Ref<Map<string, RunningTunnelHandle>>` to track running tunnels:

```ts
interface RunningTunnelHandle {
  readonly tunnelId: string
  readonly scope: Scope.CloseableScope   // close to kill the process
  readonly logs: Stream.Stream<LogEntry>  // from RunningTunnel.logs
}
```

### `run(ref)` implementation
1. Resolve tunnel by ref, get token
2. Create a `Scope.make()` 
3. Run `processSvc.run(token, opts)` within that scope (don't auto-close)
4. Store the `RunningTunnelHandle` in the Ref map keyed by tunnel ID
5. Fork a fiber that awaits `exitCode` and cleans up on exit

### `stop(ref)` implementation
1. Look up tunnel in the running map by ref
2. Close its scope (triggers SIGTERM finalizer in TunnelProcess)
3. Remove from the map

### `getLogs(ref)` implementation
1. Look up tunnel in the running map
2. Collect recent entries from the `logs` stream (or buffer logs into a sliding window)
3. Return as `ReadonlyArray<TunnelLogEntry>`

### Dependencies
- **Depends on TODO-4f878a2a** (stderr parsing) — `RunningTunnel.logs` must emit real `LogEntry` values for `getLogs` to work
- Without stderr parsing, `run`/`stop` can still work structurally (start/kill process), but `getLogs` will return empty

### cft service interface (from `packages/cft/src/services.ts`)
```ts
interface TunnelApi {
```
